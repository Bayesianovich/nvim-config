# 视频 04：调试与格式化

## 标题候选

- 我的 Neovim 为什么已经够我日常调试了
- DAP、Mason、Conform 在这套配置里怎么协同
- 不只会编辑，我这套 Neovim 也能调试和格式化

## 定位

- 建议时长：8-12 分钟
- 目标观众：写代码、调试代码的人
- 核心目标：展示这套配置不是只有编辑体验，也能覆盖开发闭环

## 开场钩子

> 很多人会把 Neovim 配置做成一个很会编辑文本的环境，但一到调试、格式化、语言工具就又回到别的 IDE 了。我的目标不是完全复制 IDE，而是把最常用、最值钱的那部分调试和格式化能力补齐。这一期我就讲这部分。

## 讲稿

### 第一段：调试是开发闭环的一部分

> 先看 `lua/plugins/dap.lua`。这里我接了 `nvim-dap`、`nvim-dap-ui`，再配合 `mason-nvim-dap` 做自动安装。语言相关的补充则拆到了 `lua/plugins/dap-lang.lua`。支持的核心语言是 Go、C/C++ 和 Python。对我来说，这已经覆盖了大部分我会在编辑器里直接调试的场景。

### 第二段：键位设计

> 键位也很直接。`F5` 开始或继续，`F1` 步入，`F2` 步过，`F3` 步出，`F7` 开关调试 UI，`<leader>db` 切换断点，`<leader>dB` 设条件断点。这一套键位的设计目标不是花哨，而是尽量接近你手已经熟悉的调试动作。

### 第三段：为什么配 Mason

> 我很看重 `mason-nvim-dap` 这一层，因为它把安装和配置衔接起来了。你在 `dap.lua` 里能看到我把 `delve`、`codelldb`、`debugpy` 放进了 `ensure_installed`。这意味着这套配置不是“你自己先把一切装好我才工作”，而是尽量把环境接线提前做掉。

### 第四段：按语言扩展

> 除了通用调试层，我还把 `dap-go` 和 `dap-python` 单独放进了 `dap-lang.lua`。这就是我扩展语言支持时的习惯：先有一个通用框架，再把确实常用的语言单独补进去，而不是一开始就把所有语言生态全装满。

### 第五段：格式化

> 再看 `lua/plugins/formatting.lua`。这里我在 `conform.nvim` 的基础上，把 C 和 C++ 的格式化明确交给 `clang-format`，并且指定了 `IndentWidth`、`TabWidth` 这些参数。这个配置很适合讲一个观点：格式化不是锦上添花，而是团队协作的最低成本约束。

> 当你的编辑器能稳定地把代码拉回一个统一风格时，你的注意力就不会浪费在缩进和对齐这种低价值争论上。

### 结尾

> 所以这套配置里，调试和格式化的核心思路其实是一致的：不是追求“把所有 IDE 功能一比一搬过来”，而是把最常用、最能降低摩擦的那部分能力接上，而且入口尽量简单。

## 录屏顺序

1. 先打开 `lua/plugins/dap.lua` 和 `lua/plugins/dap-lang.lua`，快速扫一遍通用层和语言层。
2. 切到一个简单的 Python 文件，放一个断点。
3. 演示 `F5` 启动调试，`F2`、`F1`、`F3` 走几步，`F7` 打开或关闭调试 UI。
4. 演示 `<leader>dB` 设条件断点。
5. 切到 `lua/plugins/formatting.lua`，讲 `clang-format` 配置。
6. 打开一个故意排版不整齐的 C 或 C++ 文件，执行格式化。
7. 最后回到配置文件，强调 Mason 和 Conform 的协同关系。

## 录屏前准备

- 一个能直接运行的 Python demo 文件
- 一个简单的 C 或 C++ 文件，排版故意弄乱
- 确保调试器已经装好，别把首次安装过程录进去

## 这一集要打开的文件

- `lua/plugins/dap.lua`
- `lua/plugins/dap-lang.lua`
- `lua/plugins/formatting.lua`

## 可直接念的总结

> 对我来说，调试和格式化不是额外加分项，而是开发环境的基本面。Neovim 不一定要变成一个全能 IDE，但它至少要把我每天最常用的调试路径和格式化路径打通。

## 注意事项

- 如果你录 Python 调试，尽量用最短的例子，别让观众把注意力放在业务代码上。
- 如果你录 C/C++ 格式化，提前准备一个对比明显的 before/after。
- 不建议这一集同时讲太多语言，选一个调试语言加一个格式化语言就够了。
